# 2026-06-27 修改摘要：Claude 代理抓包诊断 + 代理切 CLI 失效修复 + PTY 注入截断根治

## 一句话概述
排查了「Claude 上下行代理抓包不完整」（结论：非 bug，是生命周期），并修了同会话暴露的两个真 bug——
代理开启后切换 CLI 导致抓包失效、PTY 注入长命令时开头被吞——后者最终用「单次原子写」根治，
不再依赖任何延时定时器。

## 背景 / 目标
接着同日上一篇（[2026-06-27-Codex-approval修复与Codex-Preset支持](./2026-06-27-Codex-approval修复与Codex-Preset支持.md)）继续，
用户依次反馈三个现象，逐个排查：
1. 查看 Claude「调用详情/原始记录」时只看到 JSONL 重建（缺 system prompt + tool 定义）。
2. 先开代理、再切 claude/codex 模式后启动，代理不起效。
3. 点 ▶ 启动 claude 时命令开头被吞（如只剩 `'; set +a; … claude`），间歇出现。

## 改动清单
全部位于 `termcat_client_plugin/`，**均为未提交工作区改动**。

**Claude 代理抓包「不完整」——诊断结论：非代码 bug**
- 根因：Claude 的 JSONL 本就不含 system/tool 定义；proxy 是唯一完整来源，但只抓「启用代理后、
  且从面板 ▶ 重启」的 CLI 流量（base URL 在 launch 时注入，`ClaudeAdapter.launch`）；抓包为内存
  LRU（200 条），重载即失。即「working as designed」。验证过匹配链路无误（`turn.ts`/`captureTs`
  同为毫秒、proxy 存的是含 system+tools 的完整请求体）。
- 应对（`extension.ts`）：启用代理且检测到 CLI 在跑时，弹 warning toast 提示「需点 ▶ 重启」；
  面板加常驻 info 提示，**且带消失条件**——新增全局 `proxyEnabledAt`，当
  `captureStore.getByTimeRange(proxyEnabledAt, ∞)` 出现首条抓包即隐藏（抓到流量=已正确走代理）。

**代理开启后切 CLI 失效（真 bug，已修）**
- 根因：`ProxyServer` 的 `SseStrategy` 在 `start()` 时一次性绑定；切 CLI 只改 `selectedCliKind`、
  不重启代理（`getPort() ?? start()` 短路），strategy 永远停在最初 CLI → codex 流量打 `/responses`
  被按 Claude 的 `/v1/messages` 判定，`isCapture=false`，不抓也解析错。
- 修法：把 strategy 改为**每请求动态解析**（与已有 `getUpstreamBaseUrl` 对称）。
  `proxy-server.ts`：构造函数新增 `getStrategy: () => SseStrategy`，删 `strategy` 字段 +
  `start(strategy)` 参数，`handleRequest` 每请求 `this.getStrategy()`。`extension.ts`：构造时传入
  解析器（优先 adapter.getSseStrategy()，否则按 selectedCliKind）。`event-handlers.ts`（遗留死代码）
  同步 `start()` 无参以保编译。

**PTY 注入长命令开头被吞（真 bug，已根治）**
- 根因：`sendLine` 把 `Ctrl+U / 命令 / 回车` 作为**三次独立 write** 连发；shell 行编辑器(ZLE)
  异步消费，提示符重绘期间会丢掉命令开头（断点≈80 列折行边界）。
- 迭代过程（见下「关键决策」）：延时 → bracketed paste → **最终单次原子写**。
- 终态（`ai-code-power/src/shared/actions/pty-inject.ts` 与同步的
  `claude_code_power/src/actions/pty-inject.ts`）：`fillLine`/`sendLine` 把整条序列拼成一个字符串
  一次 `write`（`CTRL_U + text [+ '\r']`），无定时器、无 paste 标记。Windows 仍跳过 Ctrl+U。

本次无新 commit；基线 commit：`83cb4f0 ai-code-power支持codex`。

## 关键决策与理由
- **抓包不完整不当 bug 修**：核实链路后确认是生命周期+持久化所致，只做「提醒重启」的 UX 改进
  （用户在选项中选定），不强行改成"自动重启 CLP"等重方案。
- **代理 strategy 动态化 vs 切 CLI 时手动重配**：选动态解析（根因修复，连带覆盖 adapter 后到场、
  reconcile 改 kind 等情形）；否决「仅在 cliType 切换时调 start 重绑」——那是治标，漏掉 adapter 驱动的变化。
- **PTY 注入三次迭代（关键教训）**：
  1. 先加 `setTimeout` settle 延时 → **否决**：只是跟重绘赛跑，把丢失变得「可见」（开头闪现再消失），
     未消除；且依赖环境时序，不严谨。
  2. 改 bracketed paste（`\e[200~…\e[201~`）+ 单独 `\r` → **否决**：出现尾部乱码（paste 回显 +
     `\r` 抢跑 + claude 启动接管终端的渲染混合）。
  3. **单次原子写** → 采用：整串字节一次进 PTY 缓冲、按序消费，**无写入间隙可竞态**，不依赖任何延时。
  教训：定时器是治标；正确方向是**消除时序本身**（原子交付）。
- 保留 `pressKey` 里的 `gapMs`：那是 claude 对「单次 write 内连续相同转义键」去抖的必需间隔，与本问题无关。
