# 2026-06-21 修改摘要：Codex 适配器三大 Bug 修复

## 一句话概述

修复 Codex 适配器在历史记录展示、原始调用查看、实时更新三个方面的问题，核心根因是 chokidar 的 `awaitWriteFinish` 配置导致事件延迟、以及种子解析与 watcher 双重读取造成的事件重复。

## 背景 / 目标

用户报告 Codex（OpenAI CLI）会话在 ai-code-power 插件中有三个问题：

**原始问题（第一轮）**
1. 历史列表只显示第一条消息，且重复出现两次
2. 查看"原始调用"时提示"代理未捕获"（仅针对 Claude 的代理路径，无 Codex 分支）

**新问题（第二轮，第一轮修复后出现）**
1. 历史列表不显示工具数量、token 统计等 badge
2. 原始调用只显示"暂无调用记录"（`turn.apiCalls.length === 0`）
3. 第二个及后续问题不实时显示，需要关闭重新加载才可见

## 改动清单

### `src/adapters/codex/index.ts` — 修改
- `seedParse()` 由 `void` 改为返回 `number`（文件字节数），供 watcher 用作起始偏移量，消除种子与 watcher 的双重读取问题

### `src/adapters/codex/jsonl-watcher.ts` — 修改
- `start()` 新增 `initialByteOffset` 参数（默认 0），watcher 从该偏移量开始读取，避免重复处理已由种子解析的内容
- 移除 `awaitWriteFinish: {stabilityThreshold: 50, pollInterval: 20}` —— 这是实时更新延迟的根因，Codex 写事件时 chokidar 等待 50ms 无写入才触发，导致整轮回答结束后才更新
- 新增"末行不完整"保护：`handleChange` 读取新字节后向前扫描最后一个 `\n`（0x0a），仅将 `byteOffset` 推进到最后一个完整行末，避免读到写到一半的 JSON 行
- `readLines` 辅助函数替换为 `readBuffer`（返回 `Buffer` 而非 `string[]`），配合上述末行保护逻辑

### `src/adapters/codex/jsonl-parser.ts` — 修改
- 整体改写：从"扁平格式"改为"嵌套格式"，匹配实际 Codex JSONL 结构
  - 实际格式：`{type, timestamp, payload: {type, ...}}`
  - `event_msg/task_started` → `task_started`
  - `event_msg/user_message` → `user_msg`（`payload.message` 为字符串）
  - `event_msg/agent_message` → `assistant_msg`（`payload.message` 为字符串）
  - `event_msg/token_count` → `token_count`（从 `payload.info.last_token_usage` 取值）
  - `event_msg/task_complete`, `turn_aborted` → `task_complete`
  - `response_item/function_call` → `tool_call`（OpenAI 函数调用格式）
  - `response_item/function_call_output` → `tool_result`
  - `response_item/custom_tool_call` → `tool_call`（Codex 内置工具如 `apply_patch`）
  - `response_item/message role=assistant` → `assistant_msg`（从 `content[].output_text` 提取）
  - 其他类型（`reasoning`, `web_search_call`, `turn_context` 等）→ null

### `src/adapters/codex/session-index.ts` — 修改
- 新建 `CodexSessionIndex` 类，从 `CodexEvent[]` 线性重建 `UnifiedPromptTurn[]`
- `task_started` 开启新 turn，`user_msg` 设置 prompt，`assistant_msg` 更新 pending text，`tool_call` 追加 pending toolCalls，`token_count` 触发 `flushPending()` 创建 `ApiCallEntry`，`task_complete` 也触发一次 flush
- `getPromptTurns()` 末尾兜底 `flushPending()`，确保 in-progress turn 也有输出（tokenUsage 为零但 toolCalls/assistantText 有内容时即创建条目）
- 使用 `turnsCache` 缓存结果，`addEvents()` 时清空缓存

### `src/detector/process-watcher.ts` — 修改
- `queryLatestCodexThread()` 改为 `queryLatestCodexRolloutPath()`：直接查 SQLite `threads` 表的 `rollout_path` 列（已验证该列存在），省去 `resolveCodexJsonlPath()` 的路径重建逻辑
- `resolveCodexSessionFile()` 中取到 `rollout_path` 后用 `fs.promises.access()` 验证文件是否存在，不存在则 fallback 到 `findLatestCodexJsonl()`
- 删除 `CodexThread` 接口和 `resolveCodexJsonlPath()` 函数（被 `rollout_path` 直取方案替代）

### `src/extension.ts` — 修改
- 新增 `buildCodexRawResult(turn)` 函数：当 `adapter.kind === 'codex'` 时在 `buildRawResult` 中走 Codex 分支，从 `turn.apiCalls` 重建原始调用视图（工具调用 + 助手文本 + token 统计），彻底绕开"需要代理才能查看"的 Claude SSE 路径
- `turn.apiCalls.length === 0` 时显示"调用进行中，尚未收到模型响应或 token 统计，请稍后查看"，取代原来的"暂无调用记录"
- 工具 hint 提取增加 `inputObj?.cmd` 的 fallback，命中 Codex `exec_command` 工具的 `{"cmd":"..."}` 参数结构

## 关键决策与理由

### 决策 1：用 `initialByteOffset` 解决种子 + watcher 双重读取

**采用**：`seedParse()` 返回读取的字节数；`watcher.start(file, seedOffset)` 从该偏移量开始，chokidar `add` 事件触发时 `stat.size === byteOffset`，直接返回，不重读。

**否决方案**：`ignoreInitial: true`（跳过 add 事件）。被否决原因：如果 seedParse 到 watcher 启动之间文件增长了，就会漏掉这段新内容；`ignoreInitial: false` + offset 才能正确处理竞态。

### 决策 2：移除 `awaitWriteFinish`（Problem 3 根因）

**根因确认过程**：实际 Codex JSONL 数据分析显示，两次 `exec_command` 调用之间有 1-2 秒间隔（curl 等待），理论上 50ms 阈值应该会触发事件。但实测用户"第二轮不实时"说明要么 50ms 内 Codex 一直在写（快速 reasoning 事件），要么 FSEvents + `awaitWriteFinish` 存在 macOS 平台兼容性问题。两者都会被移除 `awaitWriteFinish` 解决。

**新增的末行保护**：移除 `awaitWriteFinish` 后理论上可能读到半行 JSON（虽然 Node.js 小 payload write 通常是原子的）。为安全起见，读取后向前扫 `\n`，只处理完整行，未完整的部分等下次 change 事件补全。

### 决策 3：直接用 SQLite `rollout_path` 取会话文件

**采用**：`SELECT rollout_path FROM threads WHERE cwd = ? ORDER BY updated_at_ms DESC LIMIT 1`，直接拿文件路径，无需从日期/UUID 重建。

**被替代方案**：`SELECT id, created_at_ms ... → resolveCodexJsonlPath(id, ms)` 根据日期目录 + UUID 搜索文件。被替代原因：`rollout_path` 更直接可靠，实际验证 SQLite 表中确实有该列（`PRAGMA table_info(threads)` 确认）。

### 决策 4：`buildCodexRawResult` 独立于 Claude SSE 路径

Codex 不走代理（无 SSE 流），单独实现基于 JSONL 重建的原始视图，展示 `ApiCallEntry` 的 toolCalls + assistantText + token 统计，多次调用时用 group tabs 展示。

**没有改 Claude 路径**：Claude 的代理/JSONL fallback 逻辑完全不动。

- `better-sqlite3` 在 Electron host 中不一定可用；如不可用则回落 `findLatestCodexJsonl()`（按 mtime 扫描），该回落在多 session 并存时可能返回错误文件（总是返回最近写入的 session，不一定是当前活跃的那个）。如需精确，未来可改用 Electron 内置的 `electron.remote` 访问 SQLite，或在 `session_meta` JSONL 行中找 `cwd` 字段做二次比对。
- `response_item/web_search_call` 等 Codex 特有事件目前全部返回 null，在历史列表中不体现（仅有 `exec_command` 工具调用有 badge）。
